# ダッシュボード設計 — オブジェクト指向 UI

人間向けダッシュボードの設計戦略。実装より先にこの文書を更新する(構造の変更はここで決めてからコードに落とす)。

## 責務(2つ)

| 責務 | 画面 | 問いに答える |
|---|---|---|
| **状態** | リポジトリ単票(ビルド/証拠/できごと) | 「いま何が起きているか。何が証拠つきで確認されたか」 |
| **ガイド** | この仕組みの説明書 | 「これは何の仕組みか。どんな思想で、どう動き、どんな言葉(形式言語)でできているか」 |

ナビゲーション: サイドバーの上部に「ガイド」の入口を常設する。リポジトリを選べば状態へ、ガイドを選べば説明書へ(同格の第一階層)。

## 設計の立場

- **オブジェクト指向 UI(OOUI)**: タスク(手順)ではなくオブジェクト(名詞)を UI の骨格にする。ユーザーはまず対象を選び、次に見方を選ぶ
- **ドメインモデルと 1:1**: 画面のオブジェクトはゲートのドメインモデル(life リポ os/write/ios-domain-model.md の対訳表)と同じ語で、同じ境界で切る。UI のために新しい概念を発明しない
- **ID は機械の名前、人間には人間の手がかり**: ビルドID・証拠ID は照合のための機械的な同一性。人間の一次的な手がかりは「いつ・どのスキーム・どのコミット・何が写っているか」。ID は降格して(小さく・二次的に)常に確認可能にはしておく

## 1. オブジェクトの抽出

| オブジェクト | 人間にとっての意味 | 人間向けの「顔」(一次表現) | 機械の名前(二次表現) |
|---|---|---|---|
| **リポジトリ** | 見張る対象の単位 | リポジトリ名 | repoKey(UI にほぼ出さない) |
| **ビルド** | 検証対象の成果物。「いつの・どのソースの・何のビルドか」 | 時刻 + スキーム/構成 + コミット + 未コミット変更の有無 + 色の識別点 | ビルドID(短縮・等幅・小) |
| **証拠** | 受理された観測。「何が写っているか」 | スクショ実物 + note(何を観測したか) | 証拠ID(小) |
| **できごと** | 監査ログ。「さっき何が起きたか」 | 日本語の文(ビルドを登録 / 証拠を受理 / 拒否 — 理由) | ツール名(出さない。対訳表で日本語化) |

関係: リポジトリ ⊃ ビルド ⊃ 証拠。できごとはリポジトリに属し、ビルド・証拠を参照する。
スライス2 で「完了報告」「判定」がオブジェクトに加わる予定 — ビルドと同格の第2タブとして足せる構造にしておく。

### 色の識別点(ビルドの視覚的同一性)

ビルドID の先頭16bit から色相を決定的に導出した小さな色ドットを、ビルドが登場する全ての場所(一覧・詳細・証拠カード・できごと)に付ける。人間は「青いビルド」「橙のビルド」で照合でき、ID の文字列比較を強いられない。色は補助チャネル(単独で意味を持たせない。ID 文字列は常に併記)。

## 2. ビューの設計(各オブジェクトにコレクション+シングル)

| オブジェクト | コレクションビュー | シングルビュー |
|---|---|---|
| リポジトリ | サイドバー(常設ナビ) | メイン領域全体(タブ付き) |
| ビルド | 「ビルド」タブ = マスターリスト(左) | 同タブの詳細ペイン(右): 素性 + その証拠 + そのできごと |
| 証拠 | 「証拠」タブ = ギャラリー(ビルド横断) / ビルド詳細内のグリッド | ライトボックス(オーバーレイ): 原寸スクショ + 全メタデータ + 属すビルドへのリンク |
| できごと | 「できごと」タブ = 時系列フィード | (シングルなし。行が全て) |

## 3. レイアウト構造とナビゲーション

```
┌──────────┬──────────────────────────────────────────┐
│ サイドバー   │ リポジトリ単票                                 │
│ リポジトリ   │  名前・パス・件数 + 直近の拒否があれば警告帯       │
│ コレクション │ ┌─ タブ: ビルド / 証拠 / できごと ─────────┐  │
│           │ │ [ビルド] マスター(左) ⇄ 詳細(右)          │  │
│ (全体件数   │ │ [証拠]  ギャラリー → クリックでライトボックス │  │
│  はここ)   │ │ [できごと] フィード                       │  │
│           │ └──────────────────────────────────┘  │
└──────────┴──────────────────────────────────────────┘
```

- 選択はオブジェクトの選択(リポジトリ → ビルド)。選択状態はポーリング更新をまたいで保持する
- 相互リンク: 証拠カード/ライトボックス → 属すビルドへ(ビルドタブで選択状態にして遷移)。できごとの行 → 参照するビルドへ
- 全体サマリー(旧スタットタイル行)は廃止。件数はオブジェクトの居場所に置く(サイドバーの各リポジトリ行・タブのバッジ)。「数字だけの部屋」を作らない

## 4. 表現ルール

- **ビルドの見出し**: `{scheme} {configuration} — M/d HH:mm`。ID は見出しにしない
- **ビルドの素性(詳細ペイン)**: コミット(短縮 sha・未コミット変更ありなら説明つき警告)・登録時刻・ビルドID(短縮)・.app パス
- **証拠の見出し**: note の一行目。note が無い証拠は kind 名で代替(運用上 note は書く決まり)
- **できごとは文で書く**: 「証拠を受理」「ビルドを登録(既存の記録を返却)」「証拠を拒否 — シミュレータ内は別のビルド」。ツールの英語名は出さない
- **状態の色**: 受理 = good green / 拒否 = critical red。色+アイコン+ラベルを常に併記(dataviz 検証済みパレット)。未コミット変更あり(識別子 dirty)は warning(黄)。**英語の識別子(dirty 等)を UI のラベルに出さない** — 対訳表の日本語を使う
- **時刻**: 24時間以内は相対(「14分前」)+ hover で絶対時刻、それより古いものは絶対表記
- **ライト/ダーク両対応**(prefers-color-scheme。index.html のスクリプトが html の class/data-theme に反映)

### 表現基盤(コンポーネントライブラリ)

- 見た目の基盤は **HeroUI v3**(React 19 + Tailwind CSS v4)。Tabs / Card / Chip / Button を使い、レイアウトは Tailwind ユーティリティで組む
- 状態の色は HeroUI のセマンティックカラーに割り当てる: 受理 = `success` / 拒否 = `danger` / 未コミット変更あり = `warning`。色+アイコン+ラベル併記の原則は変えない
- 固有スタイル(ビルドの色ドット・ノートのクランプ等)だけ styles.css に置く。それ以外のカスタム CSS は増やさない

## 5. ガイド画面の設計

対象読者は「この仕組みを初めて見る人」と「未来の自分」。読み物として上から下に流れる1ページ構成で、問いの順に並べる:

| 節 | 問い | 中身 | 表現 |
|---|---|---|---|
| ① これは何 | 何の仕組み? | 一文の定義: AI の「できました」に証拠を義務づける。作業は自由、「確認した」の根拠は受理された観測だけ | ヒーロー(大きな一文) |
| ② なぜ | なぜ作った? | **実際に起きた事故**から出発(A1 古いビルド誤判定 / A3 実行なき完了報告 / A7 お願いベースは破れる)。結論 = 決まりはプロンプト(お願い)ではなく型と照合(構造)で書く | 事故カード(実話) → 思想の一文 |
| ③ どう動く | どう守っている? | ループ図解: コミット → ビルド → 登録(ビルドID = 中身から計算) → 観測 → 照合して受理/拒否。原則: 実行は自由・採用は厳格 / べき等 / 記録は不変 | 番号つきステップの流れ図 |
| ④ ことば | どんな形式言語? | 命名規律(日本語が正式名・英語識別子と1:1・比喩禁止)+ 対訳表の抜粋(概念に一言説明つき)+ 判定値 OK/NG/**確認できず**(3値である理由) | 表 + カード |
| ⑤ いま | どこまでできている? | 稼働中 = スライス1(登録と照合)。次 = スライス2(完了報告・判定・合格ライン gate.yaml)。言語定義の SSOT は life リポ os/write/ | 状態リスト |

- 文体は日常語・断定・実話ベース(それっぽい一般論を書かない — 起草規律と同じ)
- **ガイドは言語定義(os/write/ios-domain-model.md)の人間向けレンダリング**。言語を変えたらガイドも同じ変更で更新する(対訳表の写しがずれたらガイドの負け)
- 機械識別子の降格ルールはガイドにも適用(英語は対訳として小さく添える)

## 6. やらないこと

- ダッシュボードからの書き込み(状態を変えるのは MCP ツールだけ)
- リアルタイム性の誇張(5秒ポーリングで十分。WebSocket はスライス2以降に必要になってから)
- repoKey・UDID 等の機械識別子の一次表示(必要な人はライトボックス/詳細の隅で見られる)
