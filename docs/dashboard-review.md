# ダッシュボード点検 — 課題と改善戦略(2026-07-22)

実施内容: dashboard/ 全ソース精読 + `~/.claude-gate` 実データ(4リポ・369イベント・34報告・47証拠)の突合 + 稼働中 UI の全タブ検分(1334px・コンソールエラー 0)。
この文書は点検結果の単独完結レポート。構造を変えると決めたら dashboard-design.md を先に改訂してから実装する(リポジトリの規律どおり)。

## 要約

症状は多いが、根本原因は 5 つに集約される。どれも「小手先の CSS 修正」ではなく、読み取りモデル(api.ts)と語彙(words.ts)の層で直すべきもの。

| # | 根本原因 | 代表症状 |
|---|---|---|
| 1 | 「不変の記録」と「今の注意」の混同 — *未解決* という導出概念がドメインに無い | 拒否 N が永久に赤 / 放置報告の積み上がり / できごと 150 件のノイズ |
| 2 | 証拠の帰属(どの報告のどの動作か)が読み取りモデルに無い | 証拠グリッドが文脈のないカードの山になる |
| 3 | 状態色と識別色の 2 体系が同じ色相空間で衝突 + 緑の過負荷 | 無意味な赤ドットが「不良」に見える / 別概念が同じ緑 |
| 4 | 語彙 1:1 が宣言だけで機構で強制されていない | 「OK/NG」と識別子露出 / ラベル表の二重管理 / 拒否文言に `compile` 生出力 |
| 5 | ログが「ツール呼び出し履歴」であって人間の読み物になっていない | report_state 二重記録(全体の 27%) / 因果の逆順 / 定型受理の洪水 |

ガイド画面は良くできている(事故カード・図解・対訳表)。問題は「ガイドで定義した語彙と状態モデルに、実 UI と実ログが追随しきれていない」ことに尽きる。

---

## 1. 記録と注意の混同(拒否が積み上がる問題の正体)

**事実(コード + 実データ)**

- サイドバーの `拒否 N` = 直近 500 イベント中の拒否の**累積数**(`api.ts overview()`)。一度上がると絶対に下がらない。claude-gate-example は 3 日前に活動停止したのに「拒否 9」が赤で出続ける — 実際は全件解決済みの監査ログ。
- できごとタブのバッジ色 = `events.some(rejected)`(App.tsx)。過去 200 件に 1 件でも拒否があれば danger のまま。
- 報告 34 件の内訳: 提出済み 16 / **合格したのに未提出 8 / 確認できず 7** / 証拠あり 2 / 下書き 1。終着していない 18 件が、終わった 16 件と同格のカードで `openedAt` 降順に混在。open 直後に放棄された報告も残る(実データ: open の 30 秒後に別報告へ移った例)。
- dashboard-design.md §3 は「**直近の**拒否があれば警告帯」と書いている。実装は累積カウント — 設計と実装が既に乖離している。

**診断**

「記録は不変(拒否も全部残る)」というゲートの設計思想は正しい。間違っているのは、UI が**記録の量をそのまま今の状態として表示している**こと。拒否はゲートが正しく働いた証で、正常ループの一部。注意に値するのは「拒否されたまま、まだ解決していない」ことだけ。この *未解決(unresolved)* という概念がドメインのどこにも定義されていないため、UI は全歴史を赤く塗るしかない。

**対策 — 読み取りモデルに「注意」を第一級で導出する**

イベントは不変のまま、api.ts の読み取りモデルに導出を足す:

- **未解決の拒否**: 拒否イベント e が未解決 ⟺ e の対象(reportId / 同一 tool+対象)に、e より後の ok(または報告状態の前進)が無い。後続の成功が拒否を「解消」する — 記録は消えない、注意だけが消える。
- **報告の現役/終着/停滞**: `submitted` = 終着。`passed` のまま一定時間提出されない・`unconfirmed` のまま人間が引き取っていない・`draft/evidenced` のまま動きが無い = 停滞(要注意)。`human_check` を宣言した報告は「人間確認待ち」として明示的に浮かせる。
- サイドバー: 赤い数字は**未解決のみ**。解消済みの拒否履歴はできごとタブの過去記録(中立色)へ降格。
- リポジトリ単票の一等地に「今見るべきもの」(未解決拒否・人間確認待ち・停滞報告)を出す。無ければ静かに「対応が必要なものはありません」。
- 完了報告タブ: 状態でグルーピング(進行中 / 人間確認待ち / 終着)。終着は既定で畳む。

これ 1 つで、拒否の積み上がり・報告の積み上がり・サイドバー赤字の意味不明さ・「今拒否があるか」の閲覧目的、が全部解ける。

## 2. 証拠の帰属欠落(グリッドが見づらい問題の正体)

**事実**

- Evidence レコードは reportId を持たない(参照は report → evidence の片方向。words.ts の設計どおり)。証拠タブはこの逆引きをしないため、カードには「ビルド + 時刻 + note」しかなく、**どの報告のどの動作を覆う証拠か**が復元できない。
- 性質の違う 3 種(縦長スクショ / テキストの check_run / 録画)を同じ auto-fill グリッドに混在 → 高さ不揃い、`object-cover` でスクショ下端切れ、**録画はポスターフレーム無しで真っ黒**、check_run は「ユニットテストをゲートが実行 — 終了コード 0」のほぼ同一カードが並び区別不能(UI 検分で確認)。

**診断**

証拠は独立した鑑賞物ではなく、報告(動作)とビルドに従属する観測。OOUI の骨格(オブジェクトの関係)は設計文書に正しく書いてあるのに、証拠のコレクションビューだけ帰属チャネルを落としている。グリッドという形式の問題ではなく、**帰属情報の欠落**が見づらさの正体。

**対策**

1. api.ts で逆引きを導出: 各証拠に「属す報告(作業名)+ 動作番号」を付けて返す(データモデルは変えない。読み取りモデルの導出で足りる)。
2. 証拠タブを kind でセグメント: 視覚証拠(スクショ/録画)= ギャラリー、確かめの記録(check_run)= リスト行(コマンド + 終了コード + headline。サムネ形式をやめる)。フィルタは 種類 / 報告 / ビルド。
3. 各カードに帰属チップ(報告名 + 動作番号 → クリックで該当カバレッジ行へ)。既にあるビルドリンクと対に。
4. 録画はポスターフレームを出す(受理時にサーバーで先頭フレームを 1 枚抽出して隣に保存するのが確実。不変コピーの原則とも整合)。

## 3. 色の二重体系の衝突

**事実(UI 実測)**

- ビルド識別ドットは buildId から 360° 全色相を導出 → 赤・橙・緑のドットが生まれ、状態色(danger/warning/success)と同じ色相が**無意味な識別色**として並ぶ。実測で 11 色確認、赤ドットのビルドは「不良」に誤読される。
- 緑の過負荷: 受理(イベント)・合格(判定)・提出済み(報告状態)という別概念が全部同じ緑。拒否の赤も 2 種類ある(サイドバー oklch h22 / イベントバッジ oklab 赤橙)。
- 状態チップ(提出済み)と分類チップ(見た目/スクショ)が同地色・同形で、文字色だけが違う → 種別が形で区別できない。

**対策 — 意味の色と同一性の色を空間ごと分離する**

- 識別ドットの色相を**状態色相の除外域**から導出する(赤 0–40° / 琥珀 60–90° / 緑 120–160° を避けた色相環サブセットに写像)。ドットに固有の形(リング)を与え「これは状態ではない」と形でも宣言する。
- チップを 3 型に分ける(§5 のコンポーネント再編と同時に): **StateChip**(意味色 + アイコン + ラベル。success/danger/warning のみ) / **TaxonomyChip**(分類。グレー枠のアウトライン形) / **IdentityDot**(識別。除外域色相 + リング形)。
- 緑の過負荷は、提出済みを「終着」の見た目(例: 塗りつぶしでなくチェック付きアウトライン)に分離するだけでも解ける。受理バッジは §5 のログ改革で登場回数自体が激減する。

## 4. 語彙の 1:1 が機構で保証されていない

**事実**

- words.ts に **VERDICT_LABEL が存在しない**。ok/ng の日本語正式名が未定義のまま、UI は「✓ OK / ✕ NG」と英語識別子を大文字化して表示。ガイドの対訳表も「OK / NG / 確認できず」と 2 英語 + 1 日本語の混成を正典化してしまっている。一方ガイド本文と報告状態は「合格 / 不合格」。同じ判定が OK / 合格 / (イベントでは)拒否、と三つ巴(UI 検分で確認)。
- `CHECK_LABEL` / `CHANGE_KIND_LABEL` が src/ios/words.ts と dashboard/src/lib.ts に**手動コピーで二重定義**。「ゲート words.ts と 1:1」というコメントはあるが、ずれても何も検知されない。`REPORT_STATE_LABEL` に至っては dashboard 側にしか無い。
- サーバー生成の拒否文言が識別子を生で出す: 「確かめ方『compile』は変更の種類『設定』の合格ラインを下回る」(open_report.ts — reason 側は `args.behaviors[].check` を生で埋め込み、fix 側だけラベル付き)。実ログに 7 件。
- ExitCodeChip / evidenceCaption の「終了コード N(赤)」— **色名を意味の語として使っている**。色+アイコン+ラベル併記の原則(色は補助チャネル)と自己矛盾。テーマや色覚によらず読める語(「失敗」)にすべき。

**診断**

「日本語が正式名・英語識別子と 1:1・ここにない語は使わない」という規律は宣言されているが、**強制する機構が無い**。対訳表のコピーが 2 か所にあり、語彙に穴(verdict)があり、サーバー文言が対訳を通らない経路を持つ。宣言と実態のずれの典型。

**対策 — words.ts を唯一の語彙にし、型で強制する**

1. words.ts に `VERDICT_LABEL`(ok/ng/unconfirmed の日本語正式名)と `REPORT_STATE_LABEL` を追加。判定値の正式名を決める(動作判定 = 「OK/NG」を正式語彙として認めるなら対訳表に明記、認めないなら「動作OK→○○」を定義。報告状態の 合格/不合格 との使い分けも対訳表で一意にする)。
2. dashboard から words.ts を**直接 import** する(同一リポジトリ。tsconfig の参照設定だけで済む)。lib.ts のラベル表コピーを全削除。ずれたら typecheck が落ちる状態にする。
3. サーバーの reason/fix 生成も全て `*_LABEL` を通す。ユーザー入力の echo(拒否理由に入る check 名)もラベル変換してから埋め込む。
4. ガイドは「言語定義の人間向けレンダリング」(設計文書 §5)なのだから、対訳表セクションは words.ts を import してレンダリングする。写しがずれる構造自体を消す。

## 5. コンポーネント層とログの物語性

**コンポーネント(現状)**

- components.tsx はあるが薄く、`SectionTitle` / `Fact` / `EvidenceThumb` が BuildsTab に、`Meta` が Lightbox に定義され、他タブがタブから import している(タブがコンポーネント庫を兼ねる層逆転)。
- 「ビルド/報告へのリンク風ボタン」がほぼ同一の className で 5 か所コピペ。muted テキスト(`text-zinc-500 dark:text-zinc-400`)が 30 か所、uppercase 見出し・ボーダー・hover パターンも散在。

**対策**: 3 層に再編する — **tokens**(意味色・muted・見出しスタイル) / **primitives**(StateChip・TaxonomyChip・IdentityDot・SectionTitle・Fact・Time・ObjectLink) / **objects**(ReportCard・EvidenceCard・EventRow・BuildRow)。atomic design を字義どおり持ち込むより、**objects をドメインオブジェクトと 1:1** にする(OOUI の写像をコンポーネント境界にも通す)方がこのリポジトリの思想に合う。styles.css が 29 行しかないのは良い状態 — 維持する。

**ログ(実データで確認した不整合)**

- `judge` 成功のたびに「判定した — 合格」+「報告が『合格』になった」が同一 ts で 2 行。submit / attach_evidence も同様。**report_state がイベント総数の 27%**(98/369)を占め、人間のキャッチアップ時に同じことを二度読ませる。
- **因果の逆順**: `report_state(証拠あり)` が原因の `attach_evidence` より**先に**記録されている実例(2026-07-21T14:26:37.009Z)。run_check より先に report_state が来る例も。監査ログとして順序が信用できない。
- 定型受理 7 連(開く→登録→受理→…→提出)が報告ごとに機械的に繰り返し、拒否・判定・人間確認という「読む価値のある行」が埋没。既定フィルタが「すべて」なのも埋没を助長。

**対策**: (a) audit の書き込み順を原因→結果に修正。(b) report_state を独立イベントでなく**原因イベントへの付記**として扱う(まず表示層で ts+reportId で 1 行に畳む: 「証拠を受理 → 報告は『証拠あり』へ」。将来は記録形式も caused フィールドに)。(c) できごとタブの既定を「節目」(拒否・判定・提出・人間確認・掃除)にし、「すべて」は選択式に。(d) 「この報告のできごと」も節目だけの要約にする。

## 6. 表示の個別課題(上の対策に含まれない残り)

- レスポンシブ: Tailwind の md:/lg: 分岐はあるが、狭幅は未検証(検分環境はウィンドウ最小 1334px)。カバレッジ表・2 ペインの狭幅挙動と、モバイルでサイドバーが畳めない点は要実機確認。
- カバレッジ表の動作説明(観測 + 修正の全文 2 段)が長大で、1 画面 1 動作しか入らない例あり。既定は 1 行要約 + 展開に。
- ビルド一覧の見出しが `whitespace-nowrap` — 長い scheme 名で溢れる。
- 相対時刻(27 分前)と絶対時刻(7/19 14:10)が同じ列に混在(24h 境界の仕様どおりだが、一覧の縦スキャンでは読み分けコストになる。列を分けるか形式を寄せる)。
- 「証拠 0」のビルドが一覧で成果ありのビルドと同格に並ぶ(注意モデル導入後は「未使用ビルド」として畳む候補)。

## 実装順(依存関係順)

1. **語彙の機構化**(§4) — 最小工数で、以後の全 UI 文言の土台になる。words.ts に穴を塞ぎ、dashboard の二重表を削除。
2. **読み取りモデルに「注意」を導出**(§1) — 拒否・報告の積み上がり問題の根本解。UI はサイドバー・単票ヘッダ・報告グルーピングが連動して変わる。
3. **証拠の帰属逆引き + 証拠タブ再設計**(§2) — 2 と同じく api.ts の導出追加が主で、UI は従属。
4. **ログの因果修正と畳み込み**(§5 ログ) — audit 書き込み順は小修正。表示の畳みは 2 の注意モデルと同時にやると自然。
5. **コンポーネント 3 層再編 + 色体系分離**(§3, §5) — 1–4 で意味論が確定してから、表現層を一括で正す(チップ 3 型・識別色相の除外域・primitives 抽出)。

着手時は dashboard-design.md を先に改訂すること(このリポジトリの規律)。§1 の「注意」と §2 の「帰属」はドメインモデルの追記になるので、オブジェクト表(§1 の表)に「注意」を導出オブジェクトとして足すところから。

## 変えなくてよいもの

- ガイド画面の構成・文体・図解(検分でも高評価。§4 の対訳生成化だけ)。
- 「記録は不変」「書き込みはツールのみ」「判定は決定論」の原則 — 今回の問題は全てこの原則の**上**の読み取りモデルで解決でき、原則自体を曲げる必要は無い。
- 拒否 → fix → 復帰のループ自体は実ログ上健全に機能している(拒否 17 件すべてに後続の成功あり。fix 文言が効いている)。
