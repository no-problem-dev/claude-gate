# ダッシュボード設計 — オブジェクト指向 UI

人間向けダッシュボードの設計戦略。実装より先にこの文書を更新する(構造の変更はここで決めてからコードに落とす)。

## 責務(2つ)

| 責務 | 画面 | 問いに答える |
|---|---|---|
| **状態** | リポジトリ単票(完了報告/ビルド/証拠/できごと) | 「いま何が起きているか。何が証拠つきで確認されたか」 |
| **ガイド** | この仕組みの説明書 | 「これは何の仕組みか。どんな思想で、どう動き、どんな言葉(形式言語)でできているか」 |

ナビゲーション: サイドバーの上部に「ガイド」の入口を常設する。リポジトリを選べば状態へ、ガイドを選べば説明書へ(同格の第一階層)。

## 設計の立場

- **オブジェクト指向 UI(OOUI)**: タスク(手順)ではなくオブジェクト(名詞)を UI の骨格にする。ユーザーはまず対象を選び、次に見方を選ぶ
- **ドメインモデルと 1:1**: 画面のオブジェクトはゲートの語彙(`src/ios/words.ts` の対訳)と同じ語で、同じ境界で切る。UI のために新しい概念を発明しない
- **ID は機械の名前、人間には人間の手がかり**: ビルドID・証拠ID は照合のための機械的な同一性。人間の一次的な手がかりは「いつ・どのスキーム・どのコミット・何が写っているか」。ID は降格して(小さく・二次的に)常に確認可能にはしておく

## 1. オブジェクトの抽出

| オブジェクト | 人間にとっての意味 | 人間向けの「顔」(一次表現) | 機械の名前(二次表現) |
|---|---|---|---|
| **完了報告** | **主役オブジェクト**。作業の「できました」の単位。判定・提出はこの中の記録 | 作業名 + 状態チップ(下書き〜提出済み)+ カバレッジ表(動作 × 証拠 × 判定)+ 提出の記録 | reportId(出さない) |
| **リポジトリ** | 見張る対象の単位 | リポジトリ名 | repoKey(UI にほぼ出さない) |
| **ビルド** | 検証対象の成果物。「いつの・どのソースの・何のビルドか」 | 時刻 + スキーム/構成 + コミット + 未コミット変更の有無 + 色の識別点 | ビルドID(短縮・等幅・小) |
| **証拠** | 受理された観測。「何が写っているか」。確かめの記録(check_run)は「何をどう実行した結果か」 | スクショ実物 + note / check_run は確かめ方 + 終了コード | 証拠ID(小) |
| **できごと** | 監査ログ。「いつ・どのオブジェクトに・何が起きたか」 | 日本語の文 + **対象オブジェクトのチップ**(報告は作業名・ビルドは色ドット) | ツール名(出さない。対訳表で日本語化) |
| **注意** | 「今、人が見るべきものは何か」。記録から毎回計算する導出(保存しない) | 未解決の拒否・人間確認待ち・提出待ち・取り込み待ちの件数と入口(注意帯・サイドバーのバッジ) | (レコードなし。読み取りモデルの純関数) |

関係: リポジトリ ⊃ {完了報告, ビルド, 証拠}。完了報告が動作ごとに証拠を参照し、証拠がビルドを参照する。
証拠の「顔」には**帰属**(どの報告のどの動作を覆う証拠か)を含める — 参照は報告→証拠の片方向だが、人間向けには読み取りモデルが逆引きして見せる。
できごとはリポジトリに属し、報告・ビルド・証拠を参照する — **できごとは独立した読み物ではなく、オブジェクトに従属する記録**。だから(a)全量はリポジトリのできごとタブに、(b)各オブジェクトのシングルビューに「そのオブジェクトのできごと」を置く、の両面で見せる。

### 色の識別点(ビルドの視覚的同一性)

ビルドID から色相を決定的に導出した小さな**リング**(輪)を、ビルドが登場する全ての場所(一覧・詳細・証拠カード・できごと)に付ける。人間は「青いビルド」「紫のビルド」で照合でき、ID の文字列比較を強いられない。色は補助チャネル(単独で意味を持たせない。ID 文字列は常に併記)。

**識別の色と意味の色は空間ごと分離する**: 色相は状態色(拒否の赤・警告の琥珀・受理の緑)の色相域を**除外した範囲**(およそ 180°〜330°: シアン〜青〜紫〜マゼンタ)に写像する。赤いドットが「不良」に誤読される事故を色空間の設計で消す。塗りつぶしではなくリングで描くのは、状態バッジと**形でも**区別するため。

### 注意の導出(未解決の拒否)

記録は不変で全部残る。しかし記録の量は「今の状態」ではない。注意(今、人が見るべきもの)は記録から毎回計算する純関数の導出で、保存しない:

- 拒否のできごとが**未解決** ⟺ その後に解消するできごとが無い:
  - 報告に紐づく拒否(reportId あり)→ 同じ報告のその後の成功(どのツールでも)で解消。
    報告が掃除で消えた場合・終着(提出済み)に達した場合も解消(終着した報告への拒否は壁であって、やることではない)
  - 報告に紐づかない拒否 → 同じツールのその後の成功で解消
- 報告のグループは状態からの導出: **人間確認待ち**(確認できず)/ **提出待ち**(合格)/ **進行中**(下書き・証拠あり・不合格)/ **終着**(提出済み)
- サイドバーのバッジ・注意帯・できごとタブのバッジは、全てこの導出だけを見る。**累積数を「今の状態」として赤く出さない** — 解消済みの拒否は監査ログの過去記録(中立)に降格する
- **注意には解決の導線を付ける**: 注意帯は件数ではなく**対象そのものへのリンク**(報告名 → 該当カードへ強調スクロール / 拒否 → できごとの拒否フィルタ)。
  人間確認待ちのカードには**人間確認フォーム**(動作の選択 + 確認内容 + 記録ボタン)を置く —
  「確認できず」は人間に渡す正式な出口だが、渡された先に記録手段が無いと注意が永遠に積み上がる
- **人間確認の入口は人間の操作面2つ + 代筆**: ①ダッシュボードのフォーム(判断材料 = 証拠を見ている場所で記録する。セッションが消えても宣言的に反映できる)②CLI `claude-gate confirm` ③セッション内で人間が「確認した」と明言したときのエージェントによる代筆(判断者は常に人間)。
  3経路とも同じべき等コア(confirmBehavior。証拠ID は内容から決定)に合流するので、どこから何度記録しても状態は1つ。
  記録は証拠(kind: human_check)になり自動で再判定される。監査には入口(via)が残る
- **ずれも導出**: 人間の動きは非同期(検証は過去・確認はさっき・提出はいま。そのときローカルは別の作業中)なので、
  判定済みで未終着の報告は、検証したソース(judgment.sourceSha)と**報告の作業ブランチ先端**を読み取りモデルが毎回比較する。
  ずれていれば発生した瞬間からカードに出す: 積まれたコミットの一覧(判断材料)+ 解消の導線(取り直し推奨 / 差分確認)。
  「人間確認で合格した数分後に、提出で初めて怒られる」をなくす(2026-07-22 の実例由来)
- **提出待ちにも人間の解決導線を付ける(差分確認)**: 検証後にコミットが積まれた合格報告は、ずれが導出で見えているカードで差分確認フォームを畳まずに出す: コミット一覧を見せた上で、人間が「判定は引き続き有効」と note 必須で引き受ける。
  記録は報告に残り自動で再判定され、sourceSha がブランチ先端まで進む — 提出の記録が指す「検証されたソース」を最新に保つ。推奨はあくまで取り直し(いまの先端で再検証)で、差分確認は人間の引き受けであることを文面で明示する
- **取り込みの状態も導出**: 提出済みの報告は、提出の記録の sha が origin のデフォルトブランチの祖先か(=取り込まれたか)を読み取りモデルが毎回確かめ、カードにチップで出す:
  「デフォルトブランチに入った」(終着の静けさ)/「取り込み待ち」(人間の番 — PR 運用なら merge、main 直運用なら push)。
  取り込み待ちは注意帯にも出す(提出済みなのに世界に反映されていないものを見落とさせない)。保存はしない — 世界のいまの姿は毎回世界に聞く

## 2. ビューの設計(各オブジェクトにコレクション+シングル)

| オブジェクト | コレクションビュー | シングルビュー |
|---|---|---|
| 完了報告 | 「完了報告」タブ(先頭)= **状態グループのカード列**(人間確認待ち → 提出待ち → 進行中 → 終着。終着は既定で畳む) | カード自体: カバレッジ表 + 判定(動作ごとの理由)+ 提出の記録 + **この報告のできごと**(直近数件) |
| リポジトリ | サイドバー(常設ナビ。件数は 報告・ビルド・証拠 + 注意バッジ) | メイン領域全体(タブ付き)。ヘッダ直下に**注意帯** |
| ビルド | 「ビルド」タブ = マスターリスト(左) | 同タブの詳細ペイン(右): 素性 + その証拠 + そのできごと |
| 証拠 | 「証拠」タブ = **視覚証拠(スクショ・録画)のギャラリー + 確かめの記録・実機レポートのリスト行**。全カードに帰属チップ(報告の作業名 + 動作番号)とビルドリンク / 報告カバレッジ表・ビルド詳細内のグリッド | ライトボックス(オーバーレイ): 原寸スクショ + 全メタデータ + 属す報告・ビルドへのリンク |
| できごと | 「できごと」タブ = **日付セクションの時系列フィード + 関心の絞り込み** | (シングルなし。行が全て) |

証拠のコレクションを2形式に分けるのは種類の性質が違うから: スクショ・録画は「見る」もの(ギャラリー)、確かめの記録・実機レポートは「読む」もの(コマンド + 終了コード + サマリ行のリスト)。同じグリッドに混ぜると、高さが揃わず、テキストカードが同一の見た目で並んで区別できなくなる(実測)。録画のサムネイルは先頭フレームを静止表示する(黒画面の自動再生にしない)。

### できごとタブの構造(セクションと絞り込み)

- **日付でセクション分け**: 今日 / 昨日 / M月D日(曜)。時系列の方向感を先に与える
- **行 = 記号(受理/拒否)+ 日本語の文 + 対象オブジェクトのチップ + 時刻**。
  報告のチップは作業名(クリックで完了報告タブの該当カードへ)、ビルドのチップは色ドット + 短縮ID
  (クリックでビルドタブへ)。掃除で消えた報告のチップは出さない(文だけ残る)
- **状態の変化は独立の行にしない**: 報告の状態を動かした原因のできごとが結果を運ぶ
  (「証拠を受理 → 報告は『証拠あり』へ」を1行で)。旧形式の独立した report_state 行は、
  隣接する原因行に表示層で畳む(同じ報告・近接時刻)。二重読みと因果の逆順をなくす
- **絞り込みは関心で3つだけ**: **節目**(既定。報告を開いた・判定・提出・掃除、+ 全ての拒否)/ 拒否だけ / すべて。
  既定を節目にするのは、定型の受理(登録・証拠受理)の洪水に読む価値のある行が埋もれるから(実測: 150件中の大半が定型受理)。
  オブジェクト別の絞り込みは作らない — それは各オブジェクトのシングルビューの「そのできごと」の役目

## 3. レイアウト構造とナビゲーション

```
┌──────────┬──────────────────────────────────────────┐
│ サイドバー   │ リポジトリ単票                                 │
│ リポジトリ   │  名前・パス・件数 + 注意帯(未解決があれば)        │
│ コレクション │ ┌─ タブ: 完了報告 / ビルド / 証拠 / できごと ──┐  │
│           │ │ [完了報告] カード列(カバレッジ表 + 判定)      │  │
│ (全体件数   │ │ [ビルド] マスター(左) ⇄ 詳細(右)          │  │
│  はここ)   │ │ [証拠]  ギャラリー → クリックでライトボックス │  │
│           │ │ [できごと] フィード                       │  │
│           │ └──────────────────────────────────┘  │
└──────────┴──────────────────────────────────────────┘
```

- 選択はオブジェクトの選択(リポジトリ → ビルド)。選択状態はポーリング更新をまたいで保持する
- 相互リンク: 証拠カード/ライトボックス → 属す報告・ビルドへ(該当タブで選択/強調して遷移)。
  できごとの行 → 参照する報告へ(完了報告タブで該当カードを強調してスクロール)・参照するビルドへ
- 全体サマリー(旧スタットタイル行)は廃止。件数はオブジェクトの居場所に置く(サイドバーの各リポジトリ行・タブのバッジ)。「数字だけの部屋」を作らない
- **注意帯**: 単票ヘッダ直下。未解決の拒否・人間確認待ち・提出待ちがあれば件数と入口(クリックで該当ビューへ)を出す。
  無ければ**何も出さない** — 静けさが正常の表現。サイドバーのバッジも同じ導出を見る(未解決の拒否=赤 / 人間確認待ち=琥珀。ゼロは非表示)

## 4. 表現ルール

- **ビルドの見出し**: `{scheme} {configuration} — M/d HH:mm`。ID は見出しにしない
- **ビルドの素性(詳細ペイン)**: コミット(短縮 sha・未コミット変更ありなら説明つき警告)・登録時刻・ビルドID(短縮)・.app パス
- **証拠の見出し**: note の一行目。note が無い証拠は kind 名で代替(運用上 note は書く決まり)
- **できごとは文で書く**: 「証拠を受理」「ビルドを登録(既存の記録を返却)」「証拠を拒否 — シミュレータ内は別のビルド」。ツールの英語名は出さない
- **状態の色**: 受理 = good green / 拒否 = critical red。色+アイコン+ラベルを常に併記(dataviz 検証済みパレット)。未コミット変更あり(識別子 dirty)は warning(黄)。**英語の識別子(dirty 等)を UI のラベルに出さない** — 対訳表の日本語を使う
- **判定値の語は層で使い分ける**: 動作の判定 = **OK / NG / 確認できず**(OK・NG は日本語の日常語として正式採用)、報告の判定 = **合格 / 不合格 / 確認できず**。層の違う語を混ぜない。定義は words.ts の対訳表が唯一
- **色名を文言に使わない**: 「終了コード 1(赤)」ではなく「終了コード 1(失敗)」。色は補助チャネルであり、文はテーマ・色覚に依存せず単独で読めること
- **ラベルの写しを持たない**: UI の全ラベル(確かめ方・変更の種類・判定値・報告状態・証拠の種類)は `src/ios/words.ts` を直接 import する。対訳表のコピーが UI 側にあるとずれても検知されない — ずれたら typecheck が落ちる形だけを信用する
- **時刻**: 24時間以内は相対(「14分前」)+ hover で絶対時刻、それより古いものは絶対表記
- **ライト/ダーク両対応**(prefers-color-scheme。index.html のスクリプトが html の class/data-theme に反映)

### 表現基盤(コンポーネントライブラリ)

- 見た目の基盤は **HeroUI v3**(React 19 + Tailwind CSS v4)。Tabs / Card / Chip / Button を使い、レイアウトは Tailwind ユーティリティで組む
- 状態の色は HeroUI のセマンティックカラーに割り当てる: 受理 = `success` / 拒否 = `danger` / 未コミット変更あり = `warning`。色+アイコン+ラベル併記の原則は変えない
- **チップは3型に固定する**(役割の違うものを同じ見た目にしない):
  - **StateChip** — 状態。意味色(success/danger/warning)+ アイコン + ラベル。終着(提出済み)は塗りではなくアウトラインで「終わった静けさ」を表す
  - **TaxonomyChip** — 分類(確かめ方・変更の種類・証拠の種類)。中立のアウトライン。意味色を使わない
  - **IdentityDot** — ビルドの識別リング。状態色相の除外域から導出(§色の識別点)。意味を持たない
- **共有部品は components.tsx に集約する**: SectionTitle / Fact / ObjectLink(報告・ビルドへのリンク)/ Time / EvidenceThumb 等。タブのファイルに共有部品を定義しない(タブがコンポーネント庫を兼ねる層逆転を禁止)
- 固有スタイル(ビルドの識別リング・ノートのクランプ等)だけ styles.css に置く。それ以外のカスタム CSS は増やさない

## 5. ガイド画面の設計

対象読者は「この仕組みを初めて見る人」と「未来の自分」。読み物として上から下に流れる1ページ構成で、問いの順に並べる:

| 節 | 問い | 中身 | 表現 |
|---|---|---|---|
| ① これは何 | 何の仕組み? | 一文の定義: AI の「できました」に証拠を義務づける。作業は自由、「確認した」の根拠は受理された観測だけ | ヒーロー(大きな一文) |
| ② なぜ | なぜ作った? | **実際に起きた事故**から出発(A1 古いビルド誤判定 / A3 実行なき完了報告 / A7 お願いベースは破れる)。最初の課題だけをここに置く(非同期の発見「確認したのに提出できない」は後から来た学びなので、③の「人間の番」で説明する)。結論 = 決まりはプロンプト(お願い)ではなく型と照合(構造)で書く | 事故カード(実話) → 思想の一文 |
| ③ どう動く | どう守っている? | ループ図解: コミット → ビルド → 登録(ビルドID = 中身から計算) → 観測 → 照合して受理/拒否。原則: 実行は自由・採用は厳格 / べき等 / 記録は不変 / ゲートは世界を読むが変えない。ループの後に**人間の番**(引き取り: 確認できず・ずれ・取り込み。非同期でよい) | 番号つきステップの流れ図 + カード |
| ④ ことば | どんな形式言語? | 命名規律(日本語が正式名・英語識別子と1:1・比喩禁止)+ 対訳表の抜粋(概念に一言説明つき)+ **モデル全体図**(概念と関係の1枚グラフ)+ 判定値 OK/NG/**確認できず**(3値である理由)+ 人間の権限の形(人間確認・差分確認) | 表 + グラフ + カード |
| ⑤ いま | どこまでできている? | 稼働中 = スライス1(登録と照合)・2a(完了報告)・2b(判定・run_check・gate.yaml)・2c 掃除(forget CLI)・3(提出=記録。取り込みに向かう操作は hook が提出の記録と照合してガード)・4(人間の非同期な操作面: 人間確認・差分確認・ずれと取り込み待ちの導出)。FSM 全状態が実運用済み(状態遷移図は④の状態マシン節に置く) | 状態リスト |

- 文体は日常語・断定・実話ベース(それっぽい一般論を書かない — 起草規律と同じ)
- **ガイドは言語定義(words.ts の語彙)の人間向けレンダリング**。対訳表セクションは words.ts を import してレンダリングする(写しを持たない — 言語を変えたらガイドは自動で追随する)。概念の一言説明だけガイド側が持つ
- 機械識別子の降格ルールはガイドにも適用(英語は対訳として小さく添える)

### 図解(概念は図で見せる)

文章で説明した概念は、必ず図でも見せる。図は自前 SVG(外部ライブラリなし・ライト/ダーク追従・ラベルは対訳表の日本語のみ、識別子は小さく併記):

| 図 | 置き場 | 表すもの |
|---|---|---|
| 役割の境界図 | ② なぜ | エージェント(自由に作る)→ ゲート(厳格に受け取る)→ 人間(観て決める)の3レーンと境界線 |
| ループの遷移図 | ③ どう動く | コミット→ビルド→登録→観測→照合の主経路と、受理 / 拒否の分岐、拒否からの戻り |
| 出所照合の図 | ③ どう動く | 登録時(.app の中身→ビルドID)と受理時(シミュレータ内の実物→再計算)の2経路が「一致?」で合流し受理/拒否に分かれる |
| **モデル全体図** | ④ ことば | **ドメインモデルの1枚グラフ**: 全概念(ノード)と概念間の関係(エッジ)。この仕組みの本質はドメインモデリング(形式言語の設計)であり、プラグインやダッシュボードはその表象 — だから言語そのものの全体像を1枚で見せる |
| **状態マシン図** | ④ ことば(モデル全体図の次) | 完了報告の一生: 下書き→証拠あり→(判定)→合格/不合格/確認できず→提出済み(提出=記録・終着)。戻り(判定無効化・集め直し・人間確認)は破線。**words.ts の遷移宣言(REPORT_TRANSITIONS)のレンダリング** — 旧・手書き SVG(Diagrams.tsx の ReportStateDiagram)はこの宣言駆動の図に置き換えて廃止。ずれ・取り込み待ちは状態ではなく導出なので図には入れず、添える文が説明する |

- 状態は角丸ボックス、受理/合格 = success・拒否/不合格 = danger・確認できず = warning の色で塗り、ラベル併記
- 図とテキストは同じ内容の別表現(図だけにしか無い情報を作らない)

### モデル全体図(概念グラフ)の設計

- **データの SSOT は words.ts**。概念の台帳(`CONCEPTS`: 識別子 → 正式名・英語表記・分類)と関係(`DOMAIN_RELATIONS`: from / to / 種類 / ラベル)を語彙と同格の一級市民として words.ts に置く。from / to は概念識別子の型で制約する — 概念を改名・削除したら typecheck が落ちる(写しを持たない規律の延長)
- **関係も語彙**: 関係の種類は固定語彙(持つ / 参照する / 作る / から導出される / の一種)+ 個別ラベル(「出所」「照合する」等)。型フィールドに現れない関係(導出・生成・人間だけの操作)こそこのモデルの中心なので、型からの自動抽出はしない — 宣言が定義
- **概念の分類で色分けする**: 役割 / 記録(不変) / 語彙(分類・値) / 導出(保存しない) / 操作 / 世界(git・GitHub)。分類は TaxonomyChip と同じ中立の表現で、状態色(success/danger/warning)を使わない(グラフに「良い・悪い」は無い)
- **描画だけライブラリを使う**(「図は自前 SVG」の唯一の例外): ノード 20+ × エッジ 30+ の自動レイアウトは手書き座標では維持できない。React Flow(@xyflow/react)+ ELK.js(elkjs)で描画し、ガイド本文のバンドルを重くしないよう遅延ロードする。ノードの見た目(正式名 + 英語識別子の2段・分類色)はカスタムノードで既存の表現ルールに従う
- **読み物として静的に振る舞う**: ガイドは上から下に読むページなので、スクロールを奪わない(ホイールズーム無効・ドラッグ移動はキャンバスのみ・ノードは動かせない)。初期表示で全体が収まる(fitView)
- **密度はホバーで解く**: 概念にマウスを載せたら、その概念の近傍(直接つながる概念と関係)だけを残して他を沈める。全体の形と、読みたい一点の両立 — エッジの既定ラベルを省く決定(上)と同じ理由
- **状態マシン図も同じ扱い**: 完了報告の一生は words.ts の遷移宣言(REPORT_TRANSITIONS)が SSOT で、図はそのレンダリング。状態の色は既存ルール(合格 = success / 不合格 = danger / 確認できず = warning・終着はアウトライン)、戻りの遷移は破線

## 6. やらないこと

- ダッシュボードからの書き込みは原則しない。**例外は人間の操作3つ**: 人間確認の記録(POST /api/confirm)・差分確認の記録(POST /api/confirm-delta)・提出の記録(POST /api/submit) —
  書き込みの境界は「エージェントの語彙(MCP)」と「人間の操作」で引く。3つとも**ゲートの状態への記録**であり、世界(git・GitHub)への実行を含まない。
  それ以外の書き込み(報告・ビルド・証拠の作成、掃除)はダッシュボードに置かない
- **世界への実行ボタンを置かない**: push・merge・レビュー可能化のボタンは作らない。ゲートの実行ファイルに世界書き込みコードを持たない分離が崩れるため。
  人間の push・merge は端末や GitHub 上で行い、ダッシュボードは判断材料(提出の記録・取り込み待ち)を見せる側に徹する
- **書き込みアクションのワンクリック実行はしない**: 必ず確認ダイアログ(何が起きるかの説明 + 実行/やめる)を挟む。
  誤クリックで状態が変わる事故を防ぐ(実際に、検証エージェントの誤クリック相当の誤記録が起きた)。
  **入力欄の Enter でもダイアログを開かない** — 日本語 IME の変換確定 Enter が誤発火する(実際に起きた)。開くのはボタンだけ
- **書き込みは直列(同時に1つ)**: 非同期の書き込み(人間確認・差分確認・提出)が実行中は、
  全面のローディング(操作を受け付けない覆い + 「書き込み中…」)を出し、他の書き込みボタンも押せない。
  二重送信と、待っている間の別対象への誤操作を防ぐ。実装は共有の書き込みロック(components.tsx の
  WriteLockProvider)1本に集約し、各フォームが個別の busy を持たない。読み取り(5秒ポーリング)は止めない
- リアルタイム性の誇張(5秒ポーリングで十分。WebSocket はスライス2以降に必要になってから)
- repoKey・UDID 等の機械識別子の一次表示(必要な人はライトボックス/詳細の隅で見られる)
